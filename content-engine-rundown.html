Full technical rundown

How the Content Engine actually works

Everything you've built, end to end: what it does, how each piece works, what it uses, and how it all ties together. Written from what's actually in the code today, not the original plan docs — a few things have drifted from the spec (called out in Known gaps).

Next.js 15 · React 19 Supabase (Postgres + Auth) Claude Sonnet 4.6 + Haiku 4.5 Gemini 3.1 Flash Image MailerLite Sanity DataForSEO Tailwind v4

In one sentence: it's an automated content engine for Mogul Design Agency that picks a topic from a stored brand strategy, has Claude draft an on-brand email or blog post in your voice, routes every draft through a mandatory human approval gate, then publishes to MailerLite (email) and Sanity (blog) — with SEO research, editing tools, images, analytics, and now recurring schedules layered on top. One brand today, built so a second brand is a config change, not a rewrite.

How it all fits together

One spine of data, one pipeline every piece of content walks through.

Everything hangs off one chain of foreign keys in Postgres. Each table is one link between "what the brand believes" and "what got published":

brands
One row today (Mogul). Voice, positioning, visual identity, guidelines, SEO defaults, MailerLite/Sanity config — the whole brand brain lives here.
strategies
The current funnel definition for the brand (which CTA type fires at which stage).
icps
Ideal customer profiles: who you're writing for, their pains, objections, vocabulary.
pillars → clusters
The themes you want to own, each expanded into a hub-and-spoke set of topics (the "topic cluster" SEO model).
topics
The actual spokes — one row per piece of content, carrying its keyword, intent, funnel stage, and status. The handoff point: the pipeline reads a topic and writes its status back.
content_jobs → drafts
One job per generation request; one draft row per version (v1, v2 after a rejection, etc.).
approvals
The audit trail of every approve/reject decision.
publications
One row per (job, target) — the record that makes publishing idempotent.
performance
A time series of stats pulled back from MailerLite after send.

And the pipeline that walks that chain, for every single piece of content:

1
Pick or research a topicFrom the Content Plan tree, a quick-chat, or a full campaign interview — see Ways to start a draft.
2
Assemble the briefBrand voice, positioning, approved guidelines, the topic's (optionally DataForSEO-validated) keyword, and the resolved CTA all get stitched into one prompt.
3
Claude drafts itSonnet 4.6, extended thinking, streamed — full copy plus a complete designed HTML document, in one call.
4
Automatic quality checksBanned terms, keyword placement, contrast, length — some by a second cheap Claude pass, some by plain code. Never blocks generation, only flags issues.
✓
You review itApprove, click-to-edit anything, or reject with feedback for a redo. Nothing ever skips this. It is enforced in code, not just the UI.
5
PublishOne click sends the approved draft to MailerLite or creates a Sanity draft document. Safe to retry — it can never double-post.
6
MeasurePull real open/click stats back whenever you want. (Feeding this back into topic selection is the one piece not built yet.)
The one rule that never bends: nothing publishes without you explicitly clicking Approve, then explicitly clicking Publish/Send. There is no auto-publish path anywhere in the code — even the new recurring schedules (§ Automation) only auto-generate, never auto-send.

Three ways to start a draft

Different front doors, same pipeline underneath.

Quick chat on Home fastest

The chat embedded right on your Home screen. Say what you want in plain language ("let's create an email for retractable banners"); it locks in a goal/key message/angle, spins up a topic if needed, and generates on your go-ahead. Backed by /api/create/chat.

New Campaign most thorough

Create → a dedicated strategic interview (/campaigns/new). Can also notice things you say about your voice mid-conversation and propose adding them to your brand profile — you explicitly confirm before anything saves. Backed by /api/campaigns/chat.

Content Plan browse & pick

Create → the actual pillar → cluster → topic tree you're building topical authority around. Tap Generate on any topic directly, or use Suggest topics — Claude proposes 5–8 new topic ideas from everything it knows about the brand, shown with checkboxes, saved only for the ones you pick.

Reject & regenerate mid-flight

Not a new draft, but the fourth way content gets (re)written: on any draft awaiting review, reject with specific feedback and Claude writes a meaningfully different version addressing it — capped at 3 versions per topic before it's time to start fresh.

All four ultimately call the same /api/generate pipeline — nothing about quality or the approval gate changes based on which door you walked in through.

The brand brain

Everything downstream — every email, every blog post — is checked against this. It's the single biggest quality lever in the whole app.

Setting it up

Onboarding chat

First-run conversation, one question at a time: brand name → description → tagline → 3 differentiators → competitors → your ideal customer → voice & tone → sender name/email → whether to auto-generate a hero image on every draft. It never asks about colors, fonts, or logo — those live in the other two setup paths below. Each answer writes straight to your brand profile as you go.

Import from your website

Paste a URL (onboarding or Settings). It reads your homepage plus up to 7 linked pages (about, services, pricing, contact, portfolio — prioritized in that order), falling back to a full headless-browser render if the site is JS-only. Colors, fonts, and logo candidates are extracted with plain code, never guessed by AI; Claude can only choose from those real candidates for voice, positioning, and products — never invent a color it didn't actually see on your site. Nothing saves until you review and approve it field by field.

The Settings forms

Nine focused forms, each independent, none auto-saving: Brand basics (name, sender identity, SEO geography), Voice (voice, tone, example posts — the single biggest lever, banned terms, and your 3-slot CTA library), Positioning (description, tagline, differentiators, competitors — has a "✨ Suggest with AI" assist), Visual identity (logo, 6 brand colors, fonts, footer/legal info, social links, the auto-image toggle + style), Brand guidelines, Primary ICP, and Funnel (maps each buyer-journey stage to a CTA type).

The brand guidelines document

Two things share this name, worth separating:

  • The guidelines data — voice & tone, messaging pillars, do/don't language, CTA philosophy. Claude synthesizes a first draft from everything else on file; you edit it in a form and explicitly Save, which is the only thing that stamps it "approved" and starts injecting it into every generation prompt as the default creative direction.
  • The guidelines document (Settings → Generate brand identity) — a polished, downloadable brand deck rendered from that data plus your positioning and visual identity. This is not a new AI call — it's a pure template render, in two swappable looks: Bold Spectrum (dark, gradient-forward) or Clean Minimal (light, editorial). Download as a standalone .html file any time.

No website yet? A from-scratch identity generator

One cheap AI call picks 6 colors plus a font pairing from a curated list of 6 email-safe combinations — it can't invent an arbitrary font, only choose a key. Code then force-corrects the result for accessibility afterward: if the model's text/background pairing fails contrast, it's fixed automatically; the accent color is darkened step by step until white button text clears a safe contrast ratio.

How a topic picks its call-to-action

A short, deterministic chain, run for every single draft:

topic.funnel_stage → strategy.funnel_definition[stage].cta_type → voice_profile.cta_library[cta_type]

A topic tagged consideration, say, resolves to the cta_type your Funnel settings assigned that stage (e.g. portfolio), which resolves to whatever CTA copy you wrote for that slot in Voice settings (e.g. "See the work →"). Claude gets that as an intent, not a literal string to paste — it writes the actual button text in your voice. Missing a link in the chain just falls back to a soft "reply or learn more" invitation.

Getting your brand's look into every draft

Visual identity isn't decorative metadata — it's rendered directly into the HTML of every email and blog post.

Your logo, 6 named colors, and heading/body font strings (set in Settings → Visual identity, or extracted from your site, or generated from scratch — see above) are deterministic tokens: the templates and the AI-designed HTML both read them directly, so a color change in Settings shows up in the very next draft with no other step. Dark mode is built the same way on both paths — a single style block with a prefers-color-scheme: dark override on classed elements, so a draft still looks right if a mail client strips the rest of the CSS. The review screen's Auto/Light/Dark toggle previews (and can force) either mode without ever touching the saved draft.

Email generation & editing

One Claude call designs the whole email — copy and layout together — then three separate editing tools let you change any of it afterward.

Generating a draft

Claude (Sonnet 4.6, extended "adaptive" thinking) gets one job: return structured copy — subject, preheader, headline, 1–8 body sections, CTA — and a complete, ready-to-send HTML document, in a single forced tool call. It's designing the layout itself, not filling in a template; it's told to build an Outlook-safe table layout, tag every region (header/eyebrow/headline/body/cta/footer) so it can be click-edited later, and bake in the dark-mode CSS. Length and section count are held to a target range that varies by email type (a promotional blast: 120–250 words; a full newsletter: 300–700) — enforced both in the prompt and by a code check that will re-ask Claude once if it comes in short.

If the model's HTML fails a sanity check (missing structure, no dark-mode CSS, or contains something it shouldn't like a script tag), the app quietly falls back to one of 3 hand-built code templates for that email's type instead — so a bad design pass never ships a broken email.

Editing a draft — three tools, one hotspot system

The preview is the editor. Every tagged region gets an invisible hotspot; tap one and a sheet opens with three tabs:

ToolWhat it doesModel used
WordingType exact replacement text, or ask Claude to rephrase this one spotHaiku, retried once, escalates to Sonnet only if both attempts fail to find a safe match
DesignColor, background, spacing, font size, alignment, bold — instant, native controlsnone — pure code
StyleFuzzy free-text asks like "make it bolder" or a suggestion chipsame Haiku → retry → Sonnet ladder

Every AI-assisted edit works by find-and-replace on the actual HTML: the model returns the exact text to find and what to put instead, and the code refuses to apply anything ambiguous — if that text appears more than once in the email and the model didn't say "replace all of them" on purpose, it errors instead of guessing which one you meant. One shared, server-saved Undo stack (10 steps deep) covers every edit type, so undo survives a page reload.

Redesign is the bigger hammer: regenerate the entire visual design from your existing copy in one shot — useful when a color change is baked into fifty little spots at once. Optionally steer it first ("darker, purple accent"). It never touches your words, only the design.

Hero images

Opt in per draft, or set a brand-wide preference to auto-generate one on every first draft. By default Haiku writes the actual image scene description first (seeing any reference image you attached), then Google's Gemini 3.1 Flash Image renders it, and the result is compressed and stored in your own image bucket. Three placement choices — above the headline, below the headline, or above the button — and regenerating a draft never re-triggers a fresh image or brings back one you deliberately removed.

You control how literally the model takes you. When you type a subject, a toggle picks between AI sharpens it (Haiku adds visual detail around your description, but everything you named is a hard constraint it can't drop or swap) and Use my words exactly (your text goes to the image model word for word — only the chosen style scaffold and brand palette are added, and the Haiku call is skipped entirely, so it's also cheaper). Every generated image also saves the exact final prompt that produced it: open the image sheet and "Prompt behind the current image" shows it, editable, with a one-tap "Regenerate with this prompt" that sends your edited version verbatim.

Quality gate before it can publish

Two independent layers. First, a cheap Claude pass flags banned words, checks where your target keyword actually landed, and writes the SEO meta title/description — informational only. Second, and this is the part that actually can't be bypassed: the instant you hit Approve, the server re-scans the exact HTML you're approving (edits included) for any word on your banned list, using your live list, not a stale snapshot. If it finds one, approval is refused with a 409 until you either fix the wording or explicitly click "Approve anyway" — a genuine code-level block, not a UI suggestion.

Blog generation & editing

Same pipeline shape as email, one real difference: the editor is the article itself, and SEO checks are pure code, not a second AI call.

Generating a post

One Claude call returns title, URL slug, meta title/description, intro, 2 or more sections (heading + body), conclusion, and a call-to-action — no separate QA pass the way email gets. The post's format is auto-detected from its title and intent (no extra step, no model call):

FormatLengthJob
Pillar2,500–4,000 wordsThe definitive piece someone bookmarks
How-to1,500–2,500 wordsNumbered, actually actionable steps
Listicle1,500–2,500 wordsReal depth per item, not filler
Case study1,500–2,500 wordsProblem → approach → results (numbers) → takeaway
Thought leadership1,000–1,800 wordsA clear stance, argued, voice-forward
Landing800–1,500 wordsOutcome-led page for one specific offer

Titles starting "How to…", numbered listicle patterns ("7 Signs…"), the words "case study," and topics tagged for the top of your funnel each nudge the format automatically — or you can force one directly when generating.

Editing — click directly on the article

Unlike email, there's no separate form: the rendered preview itself is the editor. Every part of the article — headline, slug, intro, each section's heading and body, the conclusion, the call-to-action — has an invisible edit hotspot right on the live preview. Tap one, a small sheet opens pre-filled with that exact text, save, and the whole preview re-renders from the updated copy so what you see is always exactly what would publish. Page title and meta description live in a separate small "SEO details" sheet, since they're search-engine metadata, not part of the visible article.

Reject-and-regenerate works identically to email: specific feedback, capped at 3 versions, generation continues in the background while you keep working.

Why it converts to something called "Portable Text"

Sanity — your CMS — doesn't store rich content as HTML or Markdown; it needs a structured format called Portable Text. The app has its own small, hand-built converter (deliberately not Sanity's own tool, so it can be tested in isolation) that turns the constrained Markdown your posts are written in — headings, bullets, numbered lists, quotes, bold, italic, links — into that structure at publish time. Nothing more exotic than that subset is supported, by design, since that's also all the generation prompt itself is allowed to produce.

SEO & keyword research

This part isn't AI at all — it's a straight data lookup that makes the AI's brief sharper.

Every topic starts with a keyword you (or the topic-suggestion feature) typed in as a guess. Tap Research and the app fires four real-time calls to DataForSEO — search volume, difficulty score, buyer intent, and related keywords — all in parallel, in about a second. From that point on, every generation brief for that topic upgrades from a guessed keyword to something like:

"TARGET KEYWORD (DataForSEO-validated): email marketing automation (~2,400/mo searches, difficulty 34/100, commercial intent)" — plus up to 4 secondary keywords to work in naturally.

The same upgrade feeds both email and blog generation identically — one lookup, both channels benefit. Once researched, the topic shows a colored badge (green if the difficulty is comfortably within your target range, amber if it's a harder climb) right on the Content Plan.

Separately, every draft — email or blog — gets checked for whether the keyword actually landed somewhere that matters (subject/title, first paragraph, a heading) and for text-contrast accessibility (WCAG AA, 4.5:1) wherever the design sets an explicit text/background color pair.

Publishing

One approve click, one publish click. Retrying can never send it twice.

Email → MailerLite

Publish creates a real campaign, then either sends it instantly or schedules it for later (using your MailerLite account's own timezone). Every email is guaranteed to carry the required {$unsubscribe} tag — enforced three separate times before it can ever leave, at generation, at every edit, and one final hard check right at send.

Blog → Sanity

Publish creates a draft document in your Sanity studio — never live — with the hero image uploaded as a real, referenced asset. Making it live is still your call, inside Sanity itself.

You can use your own MailerLite/Sanity account (Settings → Connections, credentials encrypted at rest) or fall back to the ones in the server's own configuration — whichever you've connected wins automatically, and nothing about a saved credential is ever shown back to you, only whether one is set. Every publish is checked against a record of what's already gone out before it ever calls MailerLite or Sanity again, so a retried click, a slow connection, or a double-tap can never post the same thing twice.

Performance

A tap, not a poll — you decide when to check.

Once an email has been sent, a "Refresh stats" button on that draft pulls real numbers straight from MailerLite: sent, opens, open rate, clicks, click rate. Each refresh is appended as a new snapshot, never overwriting the last one, so you're building a real history rather than just a live number that resets. Blog posts don't have this yet — Sanity has no analytics of its own; that's designed to come from Google Search Console later, not built yet.

Recurring automation new

Set a cadence once; a draft shows up for you to review on schedule, forever.

Settings → Recurring schedules lets you set a channel (email or blog), a cadence (daily/weekly/every-2-weeks/monthly), and optionally force a specific type. A daily check picks up anything due, generates a real draft for your oldest topic that hasn't been touched yet, and leaves it awaiting review — exactly like a manually triggered draft, with the same approval gate, same editing tools, same everything. A small badge on Home tells you when a scheduled draft is sitting there waiting. If there's genuinely nothing left to write about, it skips that tick cleanly and tries again next time rather than erroring.

This never sends or publishes anything by itself — it only ever gets you to "ready for your review," the same finish line every other path in this app stops at.

Every AI model actually in use

Two Claude models do all the writing; one Gemini model does all the images.

ModelUsed for
claude-sonnet-4-6Every full draft (email & blog), redesigns, brand guidelines synthesis, website-import extraction, topic suggestions, all 3 chat flows (onboarding/campaign/quick-create), "Suggest with AI." The one model doing all the substantial writing.
claude-haiku-4-5-20251001Cheap, fast, mechanical work: the email QA pass, first attempts at click-to-edit wording/style changes, crafting the hero-image scene description, and the from-scratch color/font generator.

On-screen cost estimates use Sonnet at $3 in / $15 out per million tokens, Haiku at $1 in / $5 out, and a flat ~$0.04 per generated image — a display estimate, not your real bill.

Cost & rate protection

Three layers keep spend boring: prompt caching marks every large reusable system prompt (brand voice, guidelines, design system) cacheable, so repeat generations pay ~10% of the input price for that prefix; a per-route rate brake stops runaway loops (8 new drafts or 6 image generations per minute, with a friendly "slow down" message rather than a crash); and an optional daily budget (DAILY_SPEND_LIMIT_USD) blocks new AI work once the last 24 hours of estimated spend — the same numbers the /logs page shows — crosses your cap. Anthropic calls also retry rate-limit and overload errors automatically with backoff (4 attempts), and each Gemini image render gets one spaced retry on a transient failure, so a busy minute degrades to "slightly slower," not "failed."

The UI itself

One native-feeling shell, one design language, reused on every single screen — nothing here is one-off.

The shell

Five destinations, always in the same order: Home, Emails, Blogs, Create, Settings — a left sidebar on a wide screen, a sticky top bar and bottom tab bar on a phone, same routes either way. Home leads with the quick-create chat, then a brand-readiness checklist (only shows up while something's still missing), your at-a-glance stats, the scheduled-draft nudge when there is one, and your 5 most recent emails.

The look — "Bold spectrum"

Deliberately dark, deliberately loud in exactly one place: an ink-black canvas, raised "carbon" surfaces for every card, and a single four-color spectrum (amber → magenta → violet → cyan) that only ever appears as one accent per screen — a thin rule under a heading, a left-edge marker on the active nav item — never as a flat wash across the whole page. Home adds the one permitted exception: a soft ambient aura of the same spectrum bleeding into the black behind the top of the screen, a gradient hairline ring and halo around the create chat, and a faint brand glow under the stat tiles — light from the gradient, not the gradient itself. Hot magenta is the one interactive/focus color everywhere: buttons, links, the caret in every text field. This page you're reading right now is built in that exact same palette, on purpose.

One shared component vocabulary

Every edit surface in the app is the same sliding Sheet; every status is the same colored Badge (amber = in review, green = approved, red = rejected); every destructive action goes through the same confirm dialog; every list — topics, drafts, connections, schedules — is the same grouped row style. The draft review screen renders your actual final HTML live inside a sandboxed preview, so what you're looking at while you edit is genuinely what would publish, not an approximation of it.

What's actually tested

86 automated tests across the parts of the app that are pure logic and therefore safe to test without a real database: the blog Markdown → Portable Text converter, credential encryption, hero-image placement math, the email dark/light preview toggle, DataForSEO response normalization, the recurring-schedule cadence math, and both generation prompts' pure helper functions. Everything that touches Supabase directly is instead verified live, by hand, against the real database — which is exactly how every feature in this document got checked before shipping.

by design
